# api/ TypeScript Migration

Incremental JS‚ÜíTS migration of the `api/` backend. Goal: type safety across the parity-heavy surfaces (HTTP ‚Üî MCP ‚Üî services bags ‚Üî instructions) and a clean target for bringing the (TypeScript) provisioning broker in. See `../broker.md`.

## Decisions

- **Runtime: `tsx` everywhere for now.** Dev (nodemon `--exec tsx`), tests (`run-api-tests.js` boots via api's local `tsx`), prod (`node --import tsx`). `tsx` is a runtime **dependency** (survives `npm ci --omit=dev`); `typescript` + `@types/*` are devDeps. Precompiling to `dist/` for prod is a later, optional hardening (add a `tsconfig.build.json` + a Docker build stage then).
- **`tsconfig.json` uses `NodeNext`, NOT `bundler`.** The API runs on Node directly, and NodeNext is the mode that expects the explicit `.js` import extensions already used throughout (199/199). Do not strip those extensions.
- **Coexistence via `allowJs:true` + `checkJs:false`.** `.js` and `.ts` interoperate; only `.ts` is type-checked, so unconverted files never error-wall the gate. ~~Flip `allowJs:false` at the end.~~ **Update (Phase 4): `allowJs` stays `true`.** api/src is 100% TS, but the api imports the sibling `outreach/` package's JS in-process via relative paths (`../../../../outreach/src/*.js`), and `allowJs:false` can't resolve those (TS7016) ‚Äî a relative specifier can't be covered by an ambient `declare module`, and `outreach/` deliberately stays JS. "No stray `.js` in api/src" is enforced by `find src -name '*.js'` instead. See Phase 4.
- **Gate:** `tsc --noEmit` is api's `typecheck` script, chained into the root `typecheck` ‚Äî so `npm test` and the pre-push hook now type-check the api too, exactly like the gui.
- **Out of scope:** migrations stay `.cjs` (node-pg-migrate runs them itself); `dev/scripts/*` stay host `.js`; `api/test/*.js` stay `.js` (mostly black-box HTTP clients) ‚Äî but the api `test` runner now uses `node --import tsx` because `mcp.test.js` is **white-box**: it imports `src/` (the in-process MCP session + `closeDb`) directly.

## Gotchas

- **NodeNext `.js` extensions** ‚Äî keep them on relative imports even inside `.ts` files; that's correct, not a mistake.
- **Two entrypoints** (`index` + `mcp/server`) ‚Äî every run context handles both.
- **`process.env.X` is `string | undefined`** under `strict` ‚Äî a small typed `env.ts` is the clean fix as those sites get converted.
- **`pg` rows are `any`** ‚Äî `client.query<Row>()` per service is where most of the real typing effort goes.
- **Fastify route typing** ‚Äî routes are plugins typed `(fastify: FastifyInstance, { svc }: { svc: SvcService })`; per-route generics `fastify.METHOD<{ Params; Querystring; Body }>` carry the request shapes (Fastify coerces per the swagger `schema`, so `integer`‚Üí`number` is accurate). `request.userId`/`fastify.searchService` come from the ambient `src/types/fastify.d.ts` augmentation ‚Äî never redeclare them inline.
- **Don't drop `?.` when narrowing `err`** ‚Äî a `catch` that did `err?.message` must stay optional-chained (e.g. `(err as Error)?.message`); rewriting to a non-optional alias (`const e = err as {‚Ä¶}; e.message`) throws on a `throw null`/`throw undefined`. Erased casts preserve runtime; `?.`‚Üí`.` does not.
